# Supabaseのデータベース設計とセキュリティを理解しよう

---

## RLS（Row Level Security）って何？

**ひとことで言うと**: 「このユーザーは自分のデータしか見られない・触れない」というルールを、データベース自体に直接書き込む仕組みです。

### なぜRLSが必要か

フロントエンドのコードだけでセキュリティを実装しようとすると、問題があります。例えば「ログインしていないと見られない」という処理をJavaScriptで書いても、URLを直接入力したり、APIを直接叩いたりすることで、そのチェックを無視できてしまいます。

RLSはデータベースに直接ルールを書くので、**どんな方法でアクセスされても必ずルールが適用されます**。JavaScriptのチェックをすり抜けても、データベースのレベルで止められます。

---

## このアプリのRLS設計

### 自分のプロフィールだけ読み書きできる（profilesテーブル）

```sql
-- 「自分のIDと一致する行だけ」操作を許可する
CREATE POLICY "profiles_own" ON profiles
  FOR ALL USING (auth.uid() = id);
```

`auth.uid()` はログイン中のユーザーのIDを返します。「自分のIDと一致する行だけ」という意味になります。他の人のプロフィールは読めないし、書き換えることもできません。

### 自分の受注案件に紐づくチャットだけ見られる（chat_messagesテーブル）

```sql
CREATE POLICY "chat_messages_own" ON chat_messages
  FOR ALL USING (
    user_project_id IN (
      SELECT id FROM user_projects WHERE user_id = auth.uid()
    )
  );
```

chat_messages テーブルには直接「このユーザーのものか」を判断できるカラムがありません。そこで「自分が受注した案件（user_projects）のIDに含まれるメッセージだけ」という形で間接的に判断しています。

## テーブルの繋がり方

このアプリのテーブルは下の図のように繋がっています。矢印は「このテーブルのIDを参照している」という意味です（外部キーと言います）。

```
auth.users（Supabaseが管理するログイン情報）
    ↓
profiles（ユーザーのプロフィール情報）
    ↓
user_projects（ユーザーと案件の紐付け）
    ↓
chat_messages（チャット履歴）
submissions（コード提出）
    ↓
reviews（AIレビュー結果）
```

このように繋げることで「誰の、どの案件の、どの提出コードに対するレビューか」をたどれるようになっています。

## IDにUUIDを使う理由

このアプリのIDは `67b5c785-d56b-4da5-b07b-14c2645e4da8` のような長い文字列（UUID）になっています。なぜ `1, 2, 3...` のような連番にしないのかというと、**連番だと予測できてしまうから**です。

例えばIDが `5` だとわかったら、`4` や `6` を試せば他のユーザーのデータにアクセスできるかもしれません。UUIDはランダムに生成されるので、推測することがほぼ不可能です。

## JSONBカラムって何？

`ai_persona`（AIクライアントの情報）と `detailed_comments`（レビューコメントの配列）は JSONB というデータ型を使っています。

**通常のカラム**: 1つの値しか入れられない（例: `name` というカラムに「田中」が入る）

**JSONB**: JSON形式のデータをまるごと保存できる（例: 名前・会社名・性格・口調・絵文字をひとまとめで保存）

```json
{
  "name": "田中 優子",
  "company": "Hair & Relax YUKI",
  "personality": "親しみやすく丁寧",
  "tone": "です/ます調",
  "avatar_emoji": "💇‍♀️"
}
```

フィールドが増えてもテーブル設計を変えなくていいので、将来の変更に強い設計です。

## Supabaseクライアントの使い分け

| ファイル | どこで使う | なぜ分けているか |
|---|---|---|
| `lib/supabase/client.ts` | ブラウザで動くコンポーネント | ブラウザがCookieを自動で管理してくれる |
| `lib/supabase/server.ts` | サーバー側の処理（API Route等） | サーバーはCookieを自分で読み書きする必要がある |
| `lib/supabase/middleware.ts` | リクエストの前処理（proxy.ts） | 毎回のリクエストのたびにセッション情報を更新するため |

## よくあるミスと対処法

1. **RLSを有効にしたけどポリシーを作り忘れた** → RLSを有効にすると「デフォルト全拒否」になる。必ず「誰が何をできるか」のポリシーも設定すること
2. **自分のデータが取れなくなった** → ポリシーが間違っている可能性がある。まずRLSを一時的に無効にして動作確認し、ポリシーを見直す
3. **`service_role` キーをフロントに書いてしまった** → これは絶対NG。このキーはRLSを完全に無視してすべてのデータにアクセスできる管理者キー。サーバー側専用
4. **`anon` キーはフロントに使ってOK** → RLSで制限された範囲しかアクセスできないので安全
