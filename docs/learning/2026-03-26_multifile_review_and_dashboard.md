## 1. コード提出を複数ファイル対応にした

### なぜ変えたか
もともとはコードを1つのテキストエリアに貼り付けるだけだった。
そのため「HTMLにCSSも全部書いてる」という状態で提出するしかなく、
AIが「CSSを別ファイルにしなさい」と指摘してくる矛盾が生じていた。

### どう変えたか
CodeSandbox（オンラインエディタ）のように、**ファイルタブを複数追加できる**UIに変更した。

```
[ index.html ] [ style.css ] [ script.js ] [+ ファイルを追加]
┌──────────────────────────────────┐
│  選択中のファイルを編集できる    │
└──────────────────────────────────┘
[3ファイルをまとめてレビュー提出]
```

ファイル名から拡張子を読み取って言語を自動判定する（例：`.css` → `css`）。

### 変更したファイル

| ファイル | 変更内容 |
|---------|---------|
| `types/index.ts` | `CodeFile` 型を新規追加。`{ name, language, content }` の構造 |
| `components/workspace/SubmitPanel.tsx` | タブ形式のマルチファイルエディタに全面書き換え |
| `app/api/review/route.ts` | `code`（文字列）の代わりに `files`（配列）を受け取るよう変更 |
| `lib/ai/reviewer.ts` | 全ファイルをAIに渡してまとめてレビューするよう変更 |
| `supabase/migrations/002_submissions_add_files.sql` | `submissions` テーブルに `files JSONB` カラムを追加 |

### 学べること：「状態を親に持ち上げる（State lifting）」
`SubmitPanel` の中で管理していた「どのファイルが選ばれているか」「各ファイルの内容」などの状態は
コンポーネント内部の `useState` で管理している。
複数の子コンポーネント間で共有したい状態は親に持ち上げるのが基本パターン。

## 2. コード提出中にアニメーションを追加した

### なぜ追加したか
提出ボタンを押してもAIのレビュー中（約30〜40秒）は画面に何も変化がなく、
「フリーズした？」と感じてしまう体験だった。

### どう変えたか
提出中は以下のようなローディング画面に切り替わるようにした：

- スピナー（くるくる回るリング）
- ステップごとに変わるアイコンとメッセージ
  1. コードを解析中...
  2. 要件との照合中...
  3. レビューコメントを生成中...
  4. スコアを算出中...
- 現在のステップを示すドット

`useEffect` + `setInterval` で2.2秒ごとにステップを進める。

### 学べること：`useEffect` と非同期処理のクリーンアップ
```typescript
useEffect(() => {
  if (!loading) { setStepIndex(0); return }
  const id = setInterval(() => {
    setStepIndex((prev) => Math.min(prev + 1, REVIEW_STEPS.length - 1))
  }, 2200)
  return () => clearInterval(id)  // ← コンポーネントが消えたときにタイマーを止める
}, [loading])
```
`return () => clearInterval(id)` の部分を**クリーンアップ関数**という。
タイマーやイベントリスナーは使い終わったら必ず止めないとメモリリークになる。

## 3. 納品ルールを実務に近い形にした

### なぜ変えたか
以前は「スコア80点以上で自動的に擬似報酬獲得」というルールだった。
しかし「要修正（error）が残ったまま80点超えた」場合でも報酬が入ってしまい、
何度提出しても報酬が重複して入るバグもあった。

### どう変えたか

**納品（完了）の条件を以下に変更した：**
1. `error` 指摘が **0件**（要修正がすべて解消されている）
2. 総合スコアが **70点以上**

**重複報酬の防止：**
すでに `completed` になっている案件へ再提出しても、
DBで `alreadyCompleted` を確認してから報酬を加算するため二重取りにならない。

**レビュー結果画面に納品判定バナーを追加：**
- 要修正あり → 赤バナー「〇件の要修正があります。修正して再提出してください」
- 納品OK → 緑バナー「納品完了！擬似報酬を獲得しました」

### 変更したファイル

| ファイル | 変更内容 |
|---------|---------|
| `app/api/review/route.ts` | 納品条件を変更・重複報酬チェックを追加 |
| `components/review/ReviewPanel.tsx` | 納品判定バナーを追加 |

### 学べること：ビジネスロジックはサーバー側で守る
「80点以上なら報酬付与」のようなルールはクライアント（ブラウザ）側だけで判定してはいけない。
悪意あるユーザーが改ざんできてしまうため、**必ずAPIルート（サーバー側）で判定する**のが原則。

## 4. 難易度ごとにAIレビューの厳しさを変えた

### なぜ変えたか
juniorの課題でも seniorと同じ厳しさでレビューされると、
`error` が大量に出て永遠に納品できない状態になっていた。

### どう変えたか
プロジェクトの難易度（junior / mid / senior）をDBから取得して、
AIへのプロンプト（指示文）に難易度別のルールを組み込んだ。

| 難易度 | `error` にする基準 |
|--------|------------------|
| **junior** | アプリが動かなくなる致命的バグ・明確なセキュリティ穴のみ |
| **mid** | ↑に加えて、重大なベストプラクティス違反 |
| **senior** | 実務でPRをマージできないレベルのもの全般（厳格） |

### 変更したファイル

| ファイル | 変更内容 |
|---------|---------|
| `lib/ai/reviewer.ts` | `difficulty` を引数で受け取り、難易度別プロンプトを構築 |
| `app/api/review/route.ts` | DBからプロジェクトの `difficulty` を取得して渡す |

### 学べること：AIへの指示（プロンプト）がそのまま動作を決める
AIはコードではなく**自然言語の指示**で動く。
「juniorには優しくして」と書けば優しくなり、「厳格に」と書けば厳しくなる。
プロンプトエンジニアリング（AIへの指示の設計）もエンジニアリングの一部。

## 5. タブを切り替えてもチャット入力が消えないようにした

### なぜ起きていたか
Reactではコンポーネントが画面から消えると（アンマウント）、
そのコンポーネントが持っていた `useState` の中身もリセットされる。

チャット入力欄は `ChatPanel` コンポーネントの中に `useState` で管理されていた。
「要件定義」タブに切り替えると `ChatPanel` がアンマウント → 入力内容が消える。

```
タブ切り替え前：  ChatPanel（input = "途中まで書いた文章"）→ アンマウント → リセット
タブ切り替え後：  ChatPanel（input = ""）→ マウント → 空になっている
```

### どう直したか
**「状態を親に持ち上げる」** パターンで解決した。

`input` の状態を `ChatPanel`（子）から `WorkspaceClient`（親）に移動した。
親コンポーネントはタブを切り替えてもアンマウントされないため、入力内容が保持される。

```
WorkspaceClient（親）
├── chatInput = "途中まで書いた文章"  ← ここで管理するので消えない
├── RequirementPanel（タブ切り替えで表示/非表示）
└── ChatPanel（タブ切り替えで表示/非表示）← chatInput を props で受け取るだけ
```

### 変更したファイル

| ファイル | 変更内容 |
|---------|---------|
| `components/workspace/WorkspaceClient.tsx` | `chatInput` state を追加し `ChatPanel` に渡す |
| `components/chat/ChatPanel.tsx` | `input` / `onInputChange` を props で受け取るよう変更 |

### 学べること：どこで状態を持つかがReact設計の肝
> 「この状態は、どのコンポーネントが使う？」を考えて管理場所を決める。
- 1つのコンポーネントだけが使う → そのコンポーネント内の `useState`
- 複数のコンポーネントが使う → 共通の親コンポーネントに持ち上げる（State lifting）

## 6. ダッシュボードの完了案件が表示されないバグを修正した

### 何が起きていたか
ダッシュボードに「納品完了件数：1件」と表示されているのに、
その下の「完了した案件」一覧には何も表示されていなかった。

### なぜ起きていたか
**原因1：Supabaseのテーブル結合（JOIN）が期待通りに動いていなかった**

元のコードはSupabaseのJOIN記法を使っていた：
```javascript
.select('*, project:projects(title, difficulty, reward_amount)')
```
この記法がこのバージョンのSupabaseで正常に動作せず、
クエリが空を返していた。

**原因2：ラベルやコードの変更がブラウザに反映されていなかった**

Next.jsがページをキャッシュしていたため、コードを変えても古い画面が表示され続けていた。

### どう直したか

**JOINをやめて別々のクエリに分けた：**
1. `user_projects` を全件取得（statusで絞り込まずJSでフィルタ）
2. 完了案件の `project_id` 一覧を取得
3. `projects` テーブルから該当プロジェクトを取得
4. JS上で結合して表示

**キャッシュ問題の解決：**
```typescript
export const dynamic = 'force-dynamic'
```
これをレイアウトとページに追加することで、毎回DBから最新データを取得するようになった。
（Next.jsはデフォルトでページをキャッシュして高速化しようとする）

### 変更したファイル

| ファイル | 変更内容 |
|---------|---------|
| `app/(main)/layout.tsx` | `force-dynamic` を追加（サイドバーの報酬が古くなる問題も解決） |
| `app/(main)/dashboard/page.tsx` | `force-dynamic` を追加・クエリをJOINなしに書き直し |

### 学べること：Next.jsのキャッシュの仕組み

Next.js App Router はパフォーマンスのためにページを積極的にキャッシュする。
DBからデータを取得するページでは、古いデータが表示されてしまうことがある。

| 設定 | 動作 |
|------|------|
| デフォルト（`auto`） | できる限りキャッシュする |
| `force-dynamic` | 毎回リクエスト時にDBから取得する |
| `force-static` | 常に静的なデータを使う |

ダッシュボードのように「ユーザーごとに違うデータ」を表示するページは
`force-dynamic` にするのが適切。
