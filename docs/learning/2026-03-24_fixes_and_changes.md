## 1. サインアップ時にプロフィールが二重登録されていた問題

### 何が起きていたか
新規登録（サインアップ）をすると「プロフィールの作成に失敗しました」というエラーが出ることがあった。

### なぜ起きていたか
プロフィールの作成が**2箇所で同時に行われていた**。

- **Supabaseのトリガー**：ユーザーを登録した瞬間に自動でプロフィールを作ってくれる仕組み
- **サインアップページのコード**：登録後にプロフィールを手動で追加するコード

この2つが同時に走ったため「同じIDのデータをもう一度入れようとした（重複エラー）」が起きていた。

### どう直したか
サインアップページの手動インサートを削除した。トリガーが自動でやってくれるので不要だった。

**学び：** 「自動でやってくれる仕組み」と「手動でやるコード」が重なるとエラーになる。どちらか一方に統一することが大切。

## 2. Supabaseのテーブル構造がアプリと合っていなかった問題

### 何が起きていたか
案件生成ボタンを押すたびに500エラーが出て、毎回エラーの内容が少しずつ変わった。

### 起きた順番と原因

#### ① `reward_amount` カラムが見つからない
Supabaseのテーブルにカラム名 `reward` があったが、アプリ側は `reward_amount` という名前で保存しようとしていた。
→ **古い `reward` カラムを削除**し、`reward_amount` に統一した。

#### ② `tech_stack` カラムが見つからない
テーブルに `tech_stack`（使用技術の配列）カラムが存在しなかった。
→ **カラムを追加**した。

#### ③ RLS（行レベルセキュリティ）がINSERTを拒否していた
Supabaseには RLS という「誰がどのデータにアクセスできるか」を制御する機能がある。
テーブルには RLS が有効になっていたが、「データを追加してよい人」のルール（ポリシー）が設定されていなかった。
→ 「ログイン済みのユーザーはデータを追加できる」というポリシーを追加した。

#### ④ SELECTポリシーも不足していた
RLSが有効な場合、「読み取り」も明示的に許可しないとブロックされる。
→ 「全員が読み取れる」ポリシーを追加した。

**学び：** Supabase の RLS は「デフォルトで全部ブロック」する設計。有効にしたら必ず「誰が何をできるか」をポリシーで設定する必要がある。

## 3. Hydration Error（画面のチラつきエラー）

### 何が起きていたか
ブラウザのコンソールに長いエラーメッセージが表示され、「サーバーとクライアントのHTMLが一致しない」と警告が出ていた。

### なぜ起きていたか
**ブラウザの拡張機能（パスワードマネージャーなど）** がページに読み込まれた後、HTMLを書き換えていた。
Reactは「サーバーが作ったHTML」と「ブラウザが表示するHTML」が一致していることを確認するが、拡張機能が勝手に属性を追加するせいで「あれ、違う！」とエラーになっていた。

### どう直したか
`<body>` タグに `suppressHydrationWarning` という属性を追加した。
これは React に「このタグだけは多少の違いは無視してね」と伝えるフラグ。

**学び：** 自分のコードに問題がなくても、ブラウザの拡張機能が原因でエラーになることがある。シークレットモードで確認すると拡張機能の影響を排除できる。

## 4. `tech_stack` が null で画面がクラッシュしていた問題

### 何が起きていたか
案件カードや作業画面で `Cannot read properties of undefined (reading 'map')` というエラーが出ていた。

### なぜ起きていたか
Supabaseに手動で入れたテストデータの `tech_stack`（使用技術のリスト）が空（null）だった。
コードは `tech_stack.map(...)` のように「必ずデータがある前提」で書かれていたため、null に対してメソッドを呼び出してクラッシュした。

### どう直したか
`tech_stack ?? []` という書き方に変えた。
`??` は「左側が null または undefined なら右側を使う」という意味。null なら空の配列 `[]` を使うので、クラッシュしなくなった。

**学び：** データベースから取得したデータは null の可能性がある。特に配列は `配列 ?? []` のように安全な書き方をする習慣をつけると良い。

## 5. ワークスペースで404エラーが出ていた問題

### 何が起きていたか
案件ボードで「案件を見る」ボタンを押すと 404（ページが見つかりません）が表示されることがあった。

### なぜ起きていたか
**外部キー制約**というデータベースのルールが原因だった。

`user_projects`（ユーザーと案件の紐付けテーブル）には「`profiles` テーブルに存在するユーザーしか登録できない」というルールがある。
しかし、手動作成のアカウントやトリガーが正常に動作しなかったケースでは `profiles` にレコードがなく、INSERT が失敗していた。
INSERT が失敗するとコードが `notFound()` を呼び出し、404 になっていた。

### どう直したか
ワークスペースページを開いたタイミングで、プロフィールがなければ自動で作成する処理を追加した（upsert：あれば無視、なければ作成）。

**学び：** 外部キー制約は「参照先のデータが必ず存在する」ことを保証する仕組み。違反すると INSERT が失敗するが、エラーが静かに握りつぶされると原因がわかりにくくなる。

## 6. AIモデルの変更（生成速度の改善）

### 変更内容
案件生成に使うAIモデルを変更した。

| 変更前 | 変更後 |
|---|---|
| `claude-opus-4-6`（最高品質・低速） | `claude-haiku-4-5-20251001`（高速・軽量） |
| 約25秒 | 約3〜5秒 |

### なぜ変えたか
案件の要件定義文を生成する用途には Haiku（軽量モデル）で十分な品質が出る。
Opus（最上位モデル）は複雑な推論が必要なタスク向けで、今回のような文章生成には過剰だった。

**学び：** AIモデルは「何でも最高品質を使えばいい」わけではない。用途に合ったモデルを選ぶことでコストと速度を大幅に改善できる。

## 7. ローディングアニメーションの追加

### 変更内容
「案件を生成」ボタンを押している間、くるくる回るスピナー（ローディングアイコン）が表示されるようにした。

### なぜ必要だったか
生成中は数秒かかるが、何も変化がないと「ボタンが効いているのかわからない」という状態になる。
ユーザーに「今処理中ですよ」と伝えることで、不安なく待ってもらえるようになる。

### 実装方法
- `loading` という true/false の状態（state）をすでに持っていた
- `loading` が true のとき、Sparkles アイコンの代わりに回転するSVGアイコンを表示するよう変更した
- CSS の `animate-spin`（Tailwind CSS のクラス）で回転アニメーションを実現

**学び：** ユーザーが操作してから結果が出るまでの間に「何かが起きている」とわかるフィードバックを見せることは、良いUIの基本。
