## 1. コードエディタに行番号を追加した

### なぜ追加したか

AIレビューのコメントに「`index.html` の42行目を修正してください」と書かれるのに、
エディタに行番号がなく、ユーザーが自分で数える必要があった。

### どう実装したか

テキストエリアの左側に「行番号専用の列（ガター）」を並べるUIにした。

```
┌────┬───────────────────────────────────────┐
│  1 │ <!DOCTYPE html>                       │
│  2 │ <html lang="ja">                      │
│  3 │   <head>                              │
│  4 │     <meta charset="UTF-8">            │
│ .. │  ...                                  │
└────┴───────────────────────────────────────┘
```

**Reactでの実装イメージ：**

```tsx
// テキストの行数を数える
const lineCount = value ? value.split('\n').length : 1

// 行番号を1からlineCountまで並べる
{Array.from({ length: lineCount }, (_, i) => (
  <div key={i}>{i + 1}</div>
))}
```

`value.split('\n')` は改行文字（`\n`）で文字列を分割した配列を返す。
その `.length` が行数になる。

### 変更したファイル

| ファイル | 変更内容 |
|---------|---------|
| `components/workspace/SubmitPanel.tsx` | `CodeEditor` コンポーネントに行番号ガターを追加 |

## 2. 行番号が途中から消えるバグを修正した（1回目）

### 何が起きていたか

行番号は表示されているが、スクロールすると途中の行から行番号が消えて
残りは空白になっていた。

```
┌────┬───────────────┐
│ 10 │ <section>     │  ← 行番号あり
│ 11 │   <div>       │  ← 行番号あり
│    │     <p>...</p>│  ← 行番号なし！（消えた）
│    │   </div>      │  ← 行番号なし！
└────┴───────────────┘
```

### なぜ起きていたか

**フォントサイズの違いが原因。**

| 要素 | クラス | 1行の高さ |
|------|--------|---------|
| テキストエリア | `text-sm` (14px) × `leading-relaxed` (1.625) | 約 22.75px |
| 行番号 | `text-xs` (12px) × `leading-relaxed` (1.625) | 約 19.5px |

最初の実装では、テキストエリアのスクロール量（`scrollTop`）に合わせて
行番号を `translateY()` で上にずらす方法を使っていた。

しかし1行あたりの高さが違うため、スクロールするほどズレが蓄積する。

```
行番号（19.5px × 行数）< テキストエリア（22.75px × 行数）

100行スクロール → 325pxのズレ → 行番号が上に消える
800行スクロール → 2600pxのズレ → ほとんど消える
```

### どう直したか（1回目）

行番号のフォントサイズを `text-xs` → `text-sm` に変更して
テキストエリアと1行の高さをそろえた。

```tsx
// 修正前
<div className="text-xs font-mono leading-relaxed ...">

// 修正後
<div className="text-sm font-mono leading-relaxed ...">
```

## 3. 行番号が途中から消えるバグを修正した（2回目）

### 何が起きていたか

フォントサイズをそろえても、まだ同じ現象が起きていた。

### なぜ起きていたか

**スクロール同期の仕組み自体に問題があった。**

最初の実装は `transform: translateY()` を使っていた。
これは行番号の入ったDIVを「CSSのアニメーションで上に移動させる」方法。

```
【問題のある実装】
行番号ガター（overflow: hidden でクリップ）
  └── 行番号を全部含むDIV（translateYで上に移動）
        ├── 1
        ├── 2
        ├── ...800
        └── ...（全部レンダリング済み）
```

この仕組みで起きる問題：

1. テキストエリアの1行の実際のピクセル高さは、ブラウザの解釈によって
   22.75pxではなく22.7pxや22.8pxになることがある
2. 800行では 800 × 0.05px = 40px のズレになる
3. 行番号DIVが想定より短くなり、スクロール後に空白が見えてしまう

### どう直したか（2回目・最終版）

**`transform` をやめて `scrollTop` を直接同期する方式に変更した。**

```tsx
// 修正後のコード
function syncScroll() {
  if (gutterRef.current && textareaRef.current) {
    // テキストエリアのスクロール位置を行番号ガターにそのままコピー
    gutterRef.current.scrollTop = textareaRef.current.scrollTop
  }
}

// テキストエリアがスクロールされたら同期する
<textarea onScroll={syncScroll} ... />

// 行が増減してReactが再描画したあとも同期する
useEffect(syncScroll)
```

行番号ガター自体も `overflow-y-scroll` にして、スクロール可能にした。
ただし見た目でスクロールバーが出ないよう CSS で非表示にしている。

```tsx
<div
  ref={gutterRef}
  className="overflow-y-scroll [&::-webkit-scrollbar]:hidden"
  style={{ scrollbarWidth: 'none' }}  // Firefox用
>
```

| 設定 | 対象ブラウザ |
|------|------------|
| `scrollbarWidth: 'none'` | Firefox |
| `[&::-webkit-scrollbar]:hidden` | Chrome / Safari / Edge |

### なぜこれで直るのか

`scrollTop` はピクセル単位の「スクロール量」そのものを表す数値。
テキストエリアと行番号ガターで同じ `scrollTop` をセットすれば、
どちらも「同じピクセル分だけ上にスクロールした状態」になる。

行の高さのズレも、フォントサイズの微妙な差も関係ない。
ピクセル単位で一致するため、どんな行数でもズレない。

## まとめ

| 修正 | 原因 | 解決策 |
|------|------|--------|
| フォントサイズ統一 | `text-xs` と `text-sm` の行高さの差 | 両方を `text-sm` に統一 |
| スクロール同期の方式変更 | `translateY` はピクセル高さの微妙な差を蓄積する | `scrollTop` を直接コピーする方式に変更 |

### 学べること：DOMのスクロール操作

```
element.scrollTop  →  そのDOMの「上からのスクロール量（ピクセル）」
```

- 読み取り：`const pos = textareaRef.current.scrollTop`
- セット：`gutterRef.current.scrollTop = pos`

Reactはこのような「見た目の状態」をstateではなくrefで操作するのが基本。
`ref.current` でDOMに直接アクセスし、必要なプロパティを操作する。
