# UI改善・AIプロンプト調整まとめ（2026-03-27）

このドキュメントでは、案件ボードとAI生成まわりで行った改善をまとめます。

---

## 1. フォルダタブのカテゴリ表示が崩れる問題

### 何が起きていたか

プロジェクトカードの上部にカテゴリ名を表示するフォルダタブを実装したとき、「LP・サービスサイト」のような長い文字列が途中で切れたり（`truncate`）、不自然な位置で折り返されていた。

```
LP・サービ     ← NG（単語の途中で切れる）
スサイト
```

### 原因

タブの幅を `w-24`（96px）で固定していたため、長いカテゴリ名が入りきらなかった。

### 解決策

**`w-fit`** を使い、内容の長さに合わせてタブ幅を自動調整するようにした。

```tsx
// Before
'w-24 h-8 rounded-t-xl ... flex items-center px-3'

// After
'w-fit min-h-8 rounded-t-xl ... flex items-center px-3 py-1'
```

フォント `text-xs` + `leading-tight` で文字が自然に収まるようになった。

**ポイント:**
- `w-fit` = 子要素のサイズに合わせた幅（`width: fit-content` に相当）
- `min-h-8` = 最低限の高さを保ちつつ、2行になっても崩れない
- 固定幅 → コンテンツ依存幅に変えることで汎用性が上がる

## 2. カテゴリの刷新（実務を意識した内容へ）

### 変更前の問題

元のカテゴリ（`blog`, `portfolio`, `game` など）は学習・趣味用途が多く、フリーランスや業務委託の実際の案件とかけ離れていた。

### 変更後のカテゴリ

| 値 | 表示名 |
|---|---|
| `lp` | LP・サービスサイト |
| `ec` | ECサイト |
| `admin` | 管理画面 |
| `dashboard` | ダッシュボード |
| `auth` | 認証・会員機能 |
| `api_integration` | 外部API連携 |
| `refactoring` | コード改善 |

スタートアップ・フリーランス案件で実際によく発生するカテゴリに絞った。

### 変更が必要だったファイル

型定義・ラベル・UIの3箇所を一致させる必要がある：

```
types/index.ts          ← ProjectCategory 型を更新
lib/utils.ts            ← CATEGORY_LABELS マッピングを更新
components/board/
  GenerateProjectButton.tsx ← セレクトボックスの選択肢を更新
```

**ポイント:** TypeScript の union type で定義しているので、型と実際の値がズレるとビルドエラーになる。変更時は3ファイルをセットで修正する。

## 3. AIが生成するクライアントの多様性問題

### 何が起きていたか

案件を何件も生成すると、クライアント名が「田中美咲」「佐藤美咲」など特定の名前に偏りがちだった。

### 原因

プロンプトに名前の多様性についての指示がなかったため、モデルが学習データの頻出パターン（日本の女性名）に引っ張られていた。

### 解決策

プロンプトに明示的な多様性の指示を追加した：

```typescript
【ai_persona の多様性について】
- 名前は毎回必ず異なる人物にすること。「田中」「佐藤」「美咲」などの特定の名前に偏らないこと
- 性別・年齢層・職種をランダムに変えること
  （例：40代男性の部長、20代女性の起業家、60代男性のオーナーなど）
- avatar_emoji も人物の属性に合わせて毎回変えること（👨‍💼👩‍💼👴👨‍🍳👩‍🔬 など）
- 会社名も業種・規模感をバラエティ豊かにすること
```

**ポイント:** LLMは指示がなければ「それらしい」デフォルト値に収束しやすい。バリエーションを求めるときは「偏らないこと」「毎回変えること」と明示的に書く必要がある。

## 4. AIが存在しないリソースに言及する問題

### 何が起きていたか

生成された案件説明文に以下のような記述が含まれることがあった：

- 「Figmaデザインカンプを参照してください」
- 「参考URLはこちら：[リンク]」
- 「初心者の方にも良い学習機会です」

いずれも架空の案件なので存在しないリンクやリソースへの言及は不自然。

### 解決策

プロンプトに禁止事項を明記した：

```typescript
【description の書き方の注意事項】
- クライアントが書いた「要件定義書」として書くこと
- Figmaデザイン・デザインカンプ・参考URLなど、実際に存在しないリソースへの言及は書かない
- 納期・予算・連絡先なども実在しないため書かない
- 「良い学習機会」「初心者向け」などの教育的なコメントは絶対に書かない
- ## 背景、## 要件、## 技術仕様 の3セクションのみで構成すること
```

**ポイント:** LLMに「〜しないこと」という制約を与えるとき、理由や代替を一緒に書くと効果が高い。「書くな」だけより「代わりに〜のみにすること」と書く方がより従いやすい。

## 5. JSONパースエラーの修正

### 何が起きていたか

プロジェクト生成APIが500エラーを返すことがあった。ログを見ると `SyntaxError: Unexpected token` というJSONパースエラー。

### 原因

AIにJSONのみを返すよう指示しても、マークダウンのコードブロックで囲んで返すことがある：

````
```json
{ "title": "..." }
```
````

これをそのまま `JSON.parse()` に渡すとエラーになる。

### 解決策

パース前にコードブロック記法を除去する処理を追加：

```typescript
const raw = message.content[0].type === 'text' ? message.content[0].text : ''

// AIがmarkdownコードブロックで返すことがあるので除去する
const text = raw
  .replace(/^```(?:json)?\s*/i, '')  // 先頭の ```json または ``` を削除
  .replace(/\s*```$/, '')             // 末尾の ``` を削除
  .trim()

const parsed = JSON.parse(text)
```

同じ問題は `reviewer.ts` でも発生していたため、同様に修正した。

**ポイント:** LLMのJSON出力は100%信頼できない。本番では `try/catch` でエラーハンドリングし、パース失敗時はリトライするか適切なエラーメッセージを返す設計が望ましい。

## 6. AIレビューモデルのコスト最適化

### 変更内容

コードレビューに使用するモデルを変更した：

```typescript
// Before
model: 'claude-opus-4-6'

// After
model: 'claude-sonnet-4-6'
```

### 判断の背景

- Opus はより高精度だが、APIコストが高い
- Sonnet でもコードレビューの品質は実用上十分
- 学習・練習用途のアプリなので、コストを抑えることを優先

**ポイント:** AIを使う機能を作るときは、精度・速度・コストのトレードオフを意識する。全機能に最高スペックのモデルを使う必要はない。
